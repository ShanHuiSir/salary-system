## 关键改动
- 后端权限模型：在认证中间件中集中维护角色列表、数据读写角色、数据管理员角色和账号管理员角色，登录与 Token 校验都会拒绝无效角色或停用账号。
- 账号管理接口：新增 `/api/v1/users` 路由，支持角色列表、账号查询、账号详情、创建账号、编辑账号、重置密码和停用账号。
- 安全约束：限制 `hr_admin` 管理 `super_admin`，禁止停用当前登录账号，至少保留一个启用状态的超级管理员，并在角色变更、停用、重置密码后撤销 refresh token。
- 数据接口权限：数据查询对所有合法角色开放，新增/导入/编辑限制为薪资业务写入角色，单条删除限制为数据管理员，全量清空仅允许超级管理员。
- 前端账号管理：新增“账号管理”页面，支持搜索账号、新增/编辑账号、按角色设置权限、设置部门范围、重置密码和停用账号。
- 页面集成：登录态中透传当前用户，布局、Header 和 Sidebar 根据当前用户显示身份信息与账号管理入口。
- 文档补充：新增 `docs/account-permissions.md`，说明数据迁移原则、批量导入后的服务器公式重算、角色权限表和账号管理 API 示例。
- 数据迁移记录：保留 `imported-data/` 下的本地存储导出、去重报告、服务器导入前备份和导入结果，便于回溯数据迁移过程。

## 问题与风险
- 已发现问题：账号管理功能目前主要通过构建验证，仍需要在浏览器中完成新增、编辑、停用、重置密码等端到端操作验证。
- 潜在风险：`departmentScope` 已经纳入账号模型和页面输入，但部门级数据过滤仍属于后续扩展点，不能误认为已完成细粒度数据隔离。
- 潜在风险：历史迁移数据文件包含多份不同来源的本地存储导出和服务器备份，提交前需要确认是否全部应该进入 Git，避免把临时备份或敏感数据长期留在仓库。
- 处理建议：正式上线前补一轮角色权限用例表，至少覆盖超级管理员、HR 管理员、HR 专员、财务、部门负责人、审计/只读六类账号。
- 处理建议：若要提交迁移数据，优先保留转换报告、去重报告和必要的脱敏样例；真实薪资数据备份建议转移到受控存储。
